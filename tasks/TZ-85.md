## 1. Цель

Реализовать автоматический расчёт **плановой себестоимости товара** на основе модульной иерархии, введённой в TZ-83:

```
Товар (Product)
   └── Модули (ProductModule)
         ├── Материалы (Material.pricePerUnit × ModuleMaterial.quantity)
         └── Виды работ (WorkType.hourlyRate × ModuleWorkType.estimatedHours)

CostCalculation = Σ(материалы) + Σ(работы) + overhead%
                  = totalMaterialCost + totalLaborCost + overheadCost
```

UI flow:
```
/products → клик товар → /products/:id (4 TZ-83 секции) → секция V «Себестоимость»
                                                                  ├─ история: список снапшотов (с totalCost, createdAt, isActive)
                                                                  ├─ кнопка «Пересчитать» → POST → новая запись
                                                                  └─ клик на снапшот → диалог breakdown (материалы, работы)
```

## 2. Архитектурные решения (зафиксированы)

| # | Решение | Почему |
|---|---|---|
| 1 | **REPLACE:** CostCalculation полностью переписан с BOM/TechProcess source на ProductModule source. Bom/TechProcess остаются как legacy schemas read-only (TZ-83 их ссылаются, но новых записей не создаём). | Решение user (Q1). Избегает dual-path сложности. Legacy CostCalculation docs с `bomId` остаются в БД как historical snapshots (уже содержат flatten materials[]/labor[] subdocs). |
| 2 | **Snapshot persistence:** CostCalculation doc сохраняется в БД при каждом `POST /products/:id/cost-calculations`. Не on-demand recompute на каждый read. | Финансовый аудит требует исторической фиксации. Read = load from DB. Write = explicit user action («Пересчитать»). |
| 3 | **No auto-invalidation:** При изменении `Material.pricePerUnit` или `WorkType.hourlyRate` старые CostCalculation snapshots НЕ обновляются автоматически. Они хранят «историческую» цену на момент расчёта. Пользователь явно делает «Пересчитать» для актуального среза. | ERP pattern. Изменения цен в прошлом не retro-активны. Audit-friendly. |
| 4 | **`overrideDimensions` НЕ влияет на стоимость** (Phase A.algorithm). Стоимость материала = `Material.pricePerUnit × ModuleMaterial.quantity`. Override габаритов — для физического раскроя (геометрия заготовки), не для финотчёта. | `Material.pricePerUnit` — линейная цена per unit (не per length/width). Чтобы override влиял, нужны per-dimension price components в Material schema — out of scope TZ-85. |
| 5 | **Aggregation:** Идентичные Material (по `materialId`) в РАЗНЫХ модулях агрегируются в одну строку `materials[]` (sum quantities + total). Аналогично для WorkType. | В breakdown UI 1 строка = «все листы ЛДСП 16мм» = cleaner than 10 dups. Я уже отражал эту логику в thinker pseudocode через `Map`. |
| 6 | **DTO cleanup:** `CreateCostCalculationDto` — `bomId` и `bomVersion` ОСТАЮТСЯ как `@IsOptional` для backward compat с legacy docs, но в TZ-85A.algorithm больше не читаются. | Не ломаем существующие e2e tests которые могут POST с bomId. |
| 7 | **Язык ответов:** Currency = RUB only (per `docs/data-model.md` политике). `Material.pricePerUnit` всегда RUB. CostCalculation.totalCost хранится в RUB без конвертации. | Currency scope = out of TZ-85 per project policy. |
| 8 | **VAT:** `vatRate` из Settings не подключается в TZ-85 расчёт. `overheadPercent` (10% default) — единственный multiplier на material cost. `vatRate` → future TZ-XX. | Решение user (Q3). Минимум изменений. |
| 9 | **Auth/Audit:** `POST /products/:id/cost-calculations` — `@Roles('admin','manager')` (полная запись recalculation). `PATCH /:id` (overheadPercent, notes) — same. `POST activate` — same. Reads (`GET`) — auth only. Audit metadata: `action: 'create'|'update'|'activate', entityType: 'CostCalculation'`. | Consistency с TZ-83A pattern (@Roles на write endpoints + @AuditAction metadata). |

## 4. Фазы реализации

### ✅ Фаза A — Backend rewrite CostCalculation (TZ-85A) — SHIPPED

**Цель:** `CostCalculationService.create()` использует ТОЛЬКО иерархию Product→Module. Bom/TechProcess legacy imports удалены. Backend typecheck exit 0.

**Подзадачи:**

1. **A.1 Drop legacy imports:** из `cost-calculation.service.ts` constructor удалить `@InjectModel(Bom.name)` и `@InjectModel(TechProcess.name)`. Добавить `@InjectModel(Product.name)`. Из top-of-file imports убрать `Bom`, `TechProcess` imports (оставить только `Product` import).
2. **A.2 Rewrite `create()` algorithm:**
   ```typescript
   async create(dto: CreateCostCalculationDto): Promise<CostCalculationDocument> {
     const productObjectId = new Types.ObjectId(dto.productId);
     const product = await this.productModel
       .findById(productObjectId)
       .populate({
         path: 'productModuleIds',
         populate: [
           { path: 'materials.materialId' },
           { path: 'workTypes.workTypeId' },
         ],
       })
       .exec();
     if (!product) throw new NotFoundException(`Product ${dto.productId} not found`);

     const matMap = new Map<string, CostMaterial>();
     const laborMap = new Map<string, CostLabor>();
     const modules = (product.productModuleIds as unknown as ProductModuleDocument[]) ?? [];
     for (const mod of modules) {
       for (const mat of mod.materials ?? []) {
         const material = mat.materialId as unknown as MaterialDocument | null;
         if (!material) continue; // orphan ref → skip silently
         const id = material._id.toString();
         const qty = mat.quantity ?? 0;
         const lineTotal = (material.pricePerUnit ?? 0) * qty;
         const prev = matMap.get(id);
         if (prev) {
           prev.quantity += qty;
           prev.total += lineTotal;
         } else {
           matMap.set(id, {
             materialId: material._id,
             materialName: material.name,
             quantity: qty,
             unit: mat.unit,
             pricePerUnit: material.pricePerUnit ?? 0,
             total: lineTotal,
           });
         }
       }
       for (const wt of mod.workTypes ?? []) {
         const workType = wt.workTypeId as unknown as WorkTypeDocument | null;
         if (!workType) continue;
         const id = workType._id.toString();
         const hours = wt.estimatedHours ?? 0;
         const lineTotal = (workType.hourlyRate ?? 0) * hours;
         const prev = laborMap.get(id);
         if (prev) {
           prev.hours += hours;
           prev.total += lineTotal;
         } else {
           laborMap.set(id, {
             workTypeId: workType._id,
             workTypeName: workType.name,
             hours,
             hourlyRate: workType.hourlyRate ?? 0,
             total: lineTotal,
           });
         }
       }
     }
     const materials = Array.from(matMap.values());
     const labor = Array.from(laborMap.values());
     const totalMaterialCost = materials.reduce((s, m) => s + m.total, 0);
     const totalLaborCost = labor.reduce((s, l) => s + l.total, 0);
     const overheadPercent = dto.overheadPercent ?? 10;
     const overheadCost = (totalMaterialCost * overheadPercent) / 100;
     const totalCost = totalMaterialCost + totalLaborCost + overheadCost;
     return this.model.create({
       productId: productObjectId,
       materials,
       totalMaterialCost,
       labor,
       totalLaborCost,
       overheadPercent,
       overheadCost,
       totalCost,
       calculatedAt: new Date(),
       notes: dto.notes,
     });
   }
   ```

## 5. Риски и tradeoff

| Риск | Вероятность | Митигация |
|---|---|---|
| Orphan Material reference (deleted after ProductModule.materials[] persisted) | Низкая | `if (!material) continue` в algorithm A.2. Orphan refs filtered out — defense in depth. |
| Material.pricePerUnit = 0 (не задана) | Средняя | Snapshot сохраняет 0. UI в breakdown: «Цена не указана» для таких строк. |
| Product.productModuleIds = [] (товар без модулей) → пустой snapshot | Средняя | UI: empty state, ясно что нужно сначала привязать модули. Backend: snapshot с all-zeros сохраняется (audit trail). |
| Race condition: пользователь меняет Material.pricePerUnit между расчётами | Низкая | Snapshot architecture по дизайну. Compensation: пользователь явно делает «Пересчитать». |
| Bom.components[].productComponentId остаётся как поле (legacy) — orphaned ref | Низкая | Bom/TechProcess out of TZ-85 scope. Frontend НЕ показывает Bom. Backend оставляет для historical legacy. |
| `workTypes.estimatedHours` или `materials.quantity` null в TZ-83 schema | Низкая | TZ-83 schema имеет `default: 1` + `default: 0`. Defensive: `?? 0` в algorithm. |
| overrideDimensions вводят в заблуждение (пользователь думает что увеличит стоимость) | Средняя | JSDoc в algorithm + UI hint в breakdown: «Override-габариты НЕ влияют на стоимость — только на физический раскрой.» |
| Нет идемпотентности для повторных POST | Средняя | Каждый POST создаёт NEW snapshot. UX: «Пересчитать» без подтверждения = OK (назн-указывает intent). Альтернатива: confirm dialog. v1 = без confirm. |
