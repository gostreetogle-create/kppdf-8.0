## 1. Symbols / Surface

> **USER-LITERAL-REQUEST vs SPEC-DEVIATION NOTE:** The user's literal request was to migrate `<select>` → `<app-pi-select>` (always-visible listbox paradigm shift). Spec §3 Decision 1 deliberately **defers** that migration (Option C: keep native `<select>` + apply pi-styling classes) because the always-visible listbox breaks form-dialog UX paradigm. This is a documented spec deviation from user-literal-request, NOT an oversight. Designer sign-off required for any deviation change (see §14 DESIGNER-1).

User reported audit: native form inputs persist across N form-dialogs despite the Pi-* UI-kit being the canonical primitive set. UI-kit adoption matrix (TZ-104 §2.1) shows:

| Primitive | Page-level adoption rate | Status |
|---|---|---|
| `<app-pi-button>` | 60+ sites | ✅ shipped |
| `<app-pi-input>` | 60+ sites | ✅ shipped |
| `<app-pi-checkbox>` | 1 site (forms kit showcase) | ❌ 0% page adoption |
| `<app-pi-select>` | 2–3 sites (forms/builder kit) | ❌ 0% page adoption |
| `<app-pi-textarea>` | 3 sites (post-TZ-104.1 commit `a2d1e84`) | ⚠️ partial — 4 broader sites |
| `<app-pi-switch>` | 4 sites (post-TZ-104.2 commit `39f269b`) | ✅ shipped |

**Three outlier primitives still span 17+ native `<select>` (TZ-104.5.3 defers), 7+ native `<input type="checkbox">` (TZ-104.5.1), and 4 broader native `<textarea>` (TZ-104.5.2) sites.**

## 2. Root cause analysis

1. **Checkbox (`<input type="checkbox">`)**: `<app-pi-checkbox>` exists but is button-based (ARIA role="checkbox"); consumers hand-roll checkboxes inline because the component lacked `id` for label-input linkage (mirrors TZ-104.1 / 39f269b a11y issue).
2. **Select (`<select>`)**: `<app-pi-select>` renders an always-visible inline listbox (Material 3 / W3C APG listbox pattern) — visually + interactively different from native `<select>` dropdown overlay. UX paradigm shift is a hard blocker for form-dialog consumers; no `pi-dropdown` primitive exists.
3. **Textarea (`<textarea>`)**: TZ-104.1 commit `a2d1e84` migrated 3 form-dialogs (products/materials/orders) but the broader 4 sites (text-block-dialog w/ markdown styling, table-template-dialog, builder-inspector, basics.page.ts) remained because:
   - `text-block-dialog.component.ts:108` uses `class="pi-input w-full font-mono text-xs"` for monospace markdown editing — pi-textarea has no `[class]` override input.
   - The remaining 3 sites are consumer of the public `pi-textarea` API but had no `[id]` wiring for `<label htmlFor>` linkage.

## 3. Architecture decisions (thinker verdict — option C on select)

| # | Decision | Verdict | Rationale |
|---|---|---|---|
| 1 | PiSelect UX | **Defer migration; apply pi-styling classes** | Always-visible listbox breaks form-dialog UX paradigm. Build new `pi-dropdown` primitive in TZ-104.6 instead. |
| 2 | PiTextarea className override | **Add `readonly customClass = input<string>('', { alias: 'class' })`** | Without this, text-block-dialog markdown styling (`font-mono text-xs`) cannot migrate. Pattern from Radix UI + Angular CDK. |
| 3 | PiCheckbox id extension | **Yes — mirror a2d1e84 / 39f269b** | a11y + e2e parity with prior TZ patterns. |
| 4 | FormArray checkbox pattern | **Per-row handler** | `[checked] = ctrl.controls.X.value`, `(checkedChange) = onToggle($index, $event)`, resolve via `array.at(idx)`. Avoids OnPush type errors. |
| 5 | Sub-task split | **TZ-104.5.1 / 104.5.2 / 104.5.3** | PR-per-primitive migration. Smaller atomic commits. |
| 6 | Sub-task execution order | **Sequential (104.5.1 → 104.5.2 → 104.5.3)** | File-set isolation. Zero conflicts. |

- `frontend/src/app/pages/organizations/organization-form-dialog.component.ts` (1 checkbox)
- `frontend/src/app/pages/materials/material-form-dialog.component.ts` (1 formArray declaration + N per-row instances: `isImmutable`)
- `frontend/src/app/pages/modules/module-materials-form-dialog.component.ts` (1 checkbox: `formControlName="isPurchased"`)

**Pattern:**
```html
<!-- Inline checkbox (work-type/product/text-block/table-template/org/materials.isActive/module-materials) -->
<app-pi-checkbox
  [checked]="form.controls.isActive.value"
  [id]="'checkbox-isActive'"
  [ariaLabel]="'Активен'"
  (checkedChange)="onActiveToggle($event)"
/>
```

```html
<!-- FormArray checkbox (materials dimensions isImmutable per row) -->
<app-pi-checkbox
  [checked]="dimGroup.controls.isImmutable.value"
  [id]="'check-immut-' + $index"
  [ariaLabel]="'Неизменяемый ' + ($index + 1)"
  (checkedChange)="onImmutableToggle($index, $event)"
/>
```

```ts
protected onActiveToggle(checked: boolean): void {
  this.form.controls.isActive.setValue(checked);
}
protected onImmutableToggle(idx: number, checked: boolean): void {
  this.dimensionsArray.at(idx).controls.isImmutable.setValue(checked);
}
```

## 12. Migration pattern summary (per primitive)

### Checkbox (TZ-104.5.1)

```html
<!-- BEFORE -->
<input id="wt-active" type="checkbox" formControlName="isActive" data-test="active-checkbox" />

<!-- AFTER -->
<app-pi-checkbox
  id="wt-active"
  [checked]="form.controls.isActive.value"
  ariaLabel="Активен"
  (checkedChange)="onActiveToggle($event)"
  data-test="active-checkbox"
/>
```

### Textarea with className override (TZ-104.5.2)

```html
<!-- BEFORE -->
<textarea
  class="pi-input w-full font-mono text-xs"
  rows="14"
  formControlName="content"
  data-test="content-input"
></textarea>

<!-- AFTER -->
<app-pi-textarea
  id="tb-content"
  formControlName="content"
  [rows]="14"
  [maxLength]="10000"
  [class]="'font-mono text-xs'"
  ariaLabel="Содержимое (CommonMark markdown)"
  data-test="content-input"
/>
```

### Native `<select>` with pi-styling (TZ-104.5.3)

```html
<!-- BEFORE -->
<select formControlName="counterpartyId" class="w-full">
  <option value="" disabled>— выберите —</option>
  @for (cp of counterparties(); track cp._id) {
    <option [value]="cp._id">{{ cp.name }}</option>
  }
</select>

<!-- AFTER (visual only — native dropdown UX) -->
<select
  formControlName="counterpartyId"
  class="pi-input w-full h-10 px-control-x text-sm hairline rounded-sm"
  [class.border-destructive]="hasError('counterpartyId')"
>
  <option value="" disabled>— выберите —</option>
  @for (cp of counterparties(); track cp._id) {
    <option [value]="cp._id">{{ cp.name }}</option>
  }
</select>
```
