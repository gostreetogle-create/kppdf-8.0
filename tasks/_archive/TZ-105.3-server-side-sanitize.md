# TZ-105.3 — Server-side DOMPurify for block HTML content

**Status:** backlog (NOT YET IMPLEMENTED)
**Parent TZ:** TZ-104.7 (Document Constructor text-block columns rendering)
**Reviewer origin:** code-reviewer-minimax-m3 verdict on TZ-104.7 final round
**Effort estimate:** ~1 day (1 backend service + 1 unit test + e2e)
**Priority:** P2 (defense-in-depth; trust model does not yet require it but should)

## Context

TZ-104.7 ships multi-column text-block rendering in the Document Constructor
via `DomSanitizer.bypassSecurityTrustHtml()` (`block-renderer.component.ts`,
see file-header trust-model banner). The trust model currently relies on:

1. **TipTap ProseMirror schema** — rejects unknown nodes/attrs at editor layer
2. **RBAC** — admin/manager-only write paths (TZ-91 whitelist)

These do **NOT** cover:

- `ImportJobsModule` bulk import paths
- `DevFixturesSeed` (and similar seeds) writing rows without going through TipTap
- Manual SQL/DB writes (admin/console-level)

Any of these paths can persist `TextBlock.column.content` with `<script>` /
`onerror=` / etc. which is then trust-bypassed and rendered.

## Goal

Add a single server-side sanitization pass at write time so that ALL paths
(in-editor AND bulk-import) converge on safe HTML.

## Scope

1. **Backend sanitization util** in `backend/src/common/sanitize-html.ts`:
   - Use `dompurify` with `jsdom` (already in dependency tree? verify).
   - Whitelist tags: TipTap StarterKit default (paragraph, heading h1-h3,
     strong, em, b, i, u, ul, ol, li, blockquote, code, pre, br, hr, a,
     span) — minimal set; add as needed.
   - Whitelist attrs: only `style` (for color/highlight), `href` (for
     links, sanitized to safe URL schemes), `data-*` (for tiptap markers).
   - Explicit deny: `<script>`, `<iframe>`, `<object>`, `<embed>`,
     on*-handlers, `javascript:` URIs.

2. **Apply at write-time** in:
   - `backend/src/modules/text-block/text-block.service.ts` — `create()`,
     `update()` for `content` and `columns[i].content`.
   - `backend/src/modules/template-block/template-block.service.ts` —
     `create()`, `update()` for `content` and `columns[i].content`.

3. **Unit test** asserting:
   - Stripped: `<script>alert(1)</script>` → ``
   - Stripped: `<a href="javascript:alert(1)">x</a>` → `<a>x</a>`
   - Stripped: `<img src=x onerror=alert(1)>` → ``
   - Kept: `<p style="color:red">hi</p>` → preserved

4. **e2e smoke test** flowing through API:
   - `POST /api/text-blocks` with malicious body → 201 + sanitized response

## Acceptance criteria

- [ ] `pnpm jest sanitize-html` passes
- [ ] `pnpm jest text-block.service` passes with new sanitization cases
- [ ] `pnpm e2e:test-text-blocks` passes (RBAC + sanitize paths)
- [ ] Trust-model banner in `block-renderer.component.ts` updated: §3
      changed from "FUTURE: NOT YET IMPLEMENTED" to "Active — `dompurify`
      pass on write (`server-side-sanitize` util)".

## Out of scope

- Changing the in-editor sanitizer approach (Angular's default stays).
- Hardening against CSS exfiltration (style allowlist narrowing) — separate
  follow-up if needed.
- Adding `dompurify` client-side (server-only is sufficient for trust model).
