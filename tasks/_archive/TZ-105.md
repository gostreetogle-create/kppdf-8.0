### 2.1 Vector DB audit — ноль присутствия

**Методика (2026-07-12 ripgrep sweep):**

```bash

   -g '*.{ts,json,md}' -g Dockerfile -g 'docker-compose*.yml'

# MongoDB Atlas vector + text search
rg '(vectorSearch|MongoDB.*\$search|\$text|\$meta|textIndex)' -g '*.ts' -g '*.json' -g '*.md'

# Embedding-семейство (в широком смысле) — поймать false positives
rg -i '(embedding|vector.?search|cosine|ANN)' -g '*.{ts,json,md}'
```

**Result:**

| Категория | Совпадений | Comment |
|---|---|---|
| Vector DB SDK / API | **0** | Нет внешних клиентов, нет Docker-сервисов, нет npm-зависимостей |
| Atlas $vectorSearch / $text | **0** | MongoDB используется как standard document store |
| Embedding-семейство | **7 false positives** | Все в `shared/theme/pi-theme-editor*` и `theme-editor.service.ts` — OKLCH color embeddings (lightness/chroma/hue channels), НЕ vector embeddings. ОКЛЧ — это perceptual color space, не поисковый индекс |

**Verdict methodology — четыре угловых risk cases (нужен ли Vector DB?):**

1. **Exact-prefix поиск (ИМЯ/ИНН/артикул/телефон)** — 100% покрыт MongoDB regex + индексы (compound). Нет смысла в semantic search для фактографических данных. **Vector DB = overkill.**
2. **MongoDB `$text` (lexical stemming index)** — built-in, available without vector DB. Dismissed because kppdf-8.0 search domain is structured-fact-lookup (counterparty name, INN, BIN, customer phone) — все запросы exact or prefix. Stemming adds cost (index-maintenance) без ROI. **`$text` не нужен.**
3. **Natural-language search по текстовым блокам документов** — TZ-86 вводит text-block конструктор, но spec не требует «найди шаблон по смыслу описания». Если появится спец-фича типа «smart block recommendation» — отдельный TZ. **Сейчас нет use-case.**
4. **Рекомендательные системы («similar products» / cross-sell)** — отсутствуют в TZ-83 + data-model.md; нет спроса от product owner. **Нет use-case.**

**Architectural verdict:** kppdf-8.0 — MongoDB-only document store + structured-name search + ad-hoc regex. Vector DB **не нужен now**, **не нужен in current roadmap**. Если semantic search появится в TZ-N+1 (future), тогда рассматривать Atlas `$vectorSearch` (если останемся на MongoDB) OR Qdrant sidecar (если MongoDB atlas level — community).

**Action (§3.1):** formalize вердикт в `ARCHITECTURE.md` секции «Search infrastructure».

### 2.2 BOM TODO — orphan ObjectId migration risk

**Текущее состояние (`backend/src/modules/bom/bom.schema.ts:11`):**

```ts
@Schema({ _id: false })
class BomComponentSchema {
  // TZ-83 cross-module side-effect: ProductComponent удалён в TZ-83 Фазе A.
  // BOM по смыслу ссылается на "продуктовую штуку" — теперь это ProductModule.
  // ref переключен на 'ProductModule'; имя поля `productComponentId` оставлено
  // чтобы не трогать DTO / frontend.
  //
  // TODO: existing boms в БД могут содержать СТАРЫЕ ObjectId из удалённой
  //       коллекции `productcomponents` — populate вернёт null. Миграция
  //       через fuzzy name-resolve в новые `productmodules` — отдельный TZ.
  //       Триггер: первое пойвление живых BOM-записей с реальными данными.
  @Prop({ type: Types.ObjectId, ref: 'ProductModule', required: true })
  productComponentId!: Types.ObjectId;
  ...
}
```

**Root cause (CONFIRMED, traced):**
- TZ-83 Фаза A удалил collection `productcomponents` и переименовал в `productmodules`.
- BOM-схема переключила `ref: 'ProductComponent'` → `ref: 'ProductModule'` для forward-совместимости.
- DTO и frontend сохранили имя поля `productComponentId` (чтобы не ломать wire format).
- **Migration риск:** существующие BOM-документы (если есть) содержат ObjectId в `components[].productComponentId`, которые валидны как IDs, но НЕ валидны как foreign keys в `productmodules`. `populate('components.productComponentId')` вернёт null, что тихо пройдёт во frontend (что сломает UI render cost-calculation / production-order).

**Severity:** MEDIUM — НЕ блокер (BOM ничтожно используется в текущей кодовой ветке — ни одного живого рабочего flow через BOM endpoint производственных цепочек), но техдолг.

**Mitigation strategy (per thinker architect verdict):**
- **Idempotent migration script** в `backend/src/modules/bom/migrations/bom-component-resolve.service.ts`.
- **Dry-run mode** через env flag `BOM_MIGRATE_DRY_RUN=true` (default в prod = dry-run = log-only).
- **Strategies per orphan (cascade):**
  1. **Direct lookup:** найти новый `ProductModule` с тем же `_id` — succeed → update ref metadata if needed; orphan resolves.
  2. **Fuzzy name resolve:** если orphan не найден по _id, ищем `ProductModule` где `name === orphan.notes` (BOM хранит человеко-читаемое имя компонента в `notes` — fallback).
  3. **Soft-detach:** если ничего не найдено, `productComponentId = null` (НЕ удаление строки!) + `notes += ' [orphan-resolved 2026-07-12]'` audit trail.
- **Не блокирует bootstrap:** ошибка в миграции → лог + exit code для monitoring; приложение стартует.
- **Trigger:** `OnApplicationBootstrap` lifecycle hook (отдельный от `OnModuleInit` чтобы не блокировать DI graph init).

**Замена TODO docstring (§3.2):** TODO строка удаляется из schema. Заменяется JSDoc-style комментарием «Migration handled by `BomsComponentResolveService` — 2026-07-12.»

**Service scaffold (key API shape):**

> **Lifecycle choice — `OnApplicationBootstrap` (NOT `OnModuleInit`):**
> TZ-87 archive (`tasks/_archive/2026-07/TZ-87.md.done` §B.1) established
> `OnModuleInit` as **canonical pattern for SEEDS** (idempotent skip-if-exists
> writes that bootstrap the database). This service is a **MIGRATION** —
> cross-collection reads (`boms` → `productmodules`) require the full DI
> graph resolved (both models available, transformer pipeline active).
> `OnApplicationBootstrap` runs AFTER all module init hooks complete,
> giving us guaranteed access to both collection handles. Pattern is
> canonical for MIGRATIONS; distinct from SEEDS. Documented inline.

```ts
@Injectable()
export class BomComponentResolveService implements OnApplicationBootstrap {
  constructor(
    @InjectModel(Bom.name) private readonly bomModel: Model<BomDocument>,
    @InjectModel(ProductModule.name) private readonly pmModel: Model<ProductModuleDocument>,
    private readonly logger: Logger,
  ) {}

  async onApplicationBootstrap(): Promise<void> {
    const dryRun = process.env.BOM_MIGRATE_DRY_RUN !== 'false';
    try {
      await this.resolveOrphans({ dryRun });
    } catch (err) {
      this.logger.error(`bom.resolve failed: ${(err as Error).message}`);
      // intentionally do NOT throw — DO NOT BLOCK bootstrap
    }
  }

  async resolveOrphans(opts: { dryRun: boolean }): Promise<{ ...
    migrated: number; softDetached: number; skipped: number;
  }> {
    // 1. find all bom components where productComponentId DOES NOT
    //    resolve in productmodules
    // 2. cascade:
    //    a) direct match (same id still in productmodules — albeit rare
    //       post-TZ-83) → flag migrated=0 (already correct ref)
    //    b) fuzzy name match on pm.name === bom-component.notes → update
    //       (only if pm.name unique enough)
    //    c) soft-detach: productComponentId = null + notes += audit marker
    // 3. return summary
  }
}
```

**Three test cases for `*.spec.ts`:**

1. **Case-direct:** bom has productComponentId pointing to a VALID existing productmodule → resolved to 0 actions (already correct).
2. **Case-fuzzy:** bom has productComponentId pointing to a deleted productcomponent, AND `notes === 'Болт М8'` exists as `ProductModule.name === 'Болт М8'` → fuzzy-resolve succeeds, `migrated=1, softDetached=0`.
3. **Case-orphan:** bom has productComponentId pointing to a deleted productcomponent, AND no fuzzy match → soft-detach, `migrated=0, softDetached=1`.

**Env flag:** `BOM_MIGRATE_DRY_RUN=true` (default in dev + prod) — log only, no DB writes. `BOM_MIGRATE_DRY_RUN=false` — actual mutation. Documents env in `backend/README.md`.

## 10. Risks & Tradeoffs

| # | Риск | P | Митигация |
|---|------|---|-----------|
| R-1 | **TZ-105.3 conflicts with TZ-104 (still in flight)** — оба трогают те же `*.page.ts` файлы | HIGH | TZ-105.3 must START only AFTER TZ-104 branch-a + branch-b merged to main. Mandatory sequencing per §3.2 graph. |
| R-2 | **BOM migration runs in production on real legacy data** — risk of data corruption or mis-resolve | MEDIUM | Dry-run mode (`BOM_MIGRATE_DRY_RUN=true`) default in prod; log-only path; explicit opt-in to mutating. Soft-detach preserves rows (only clears orphan FK). |
| R-3 | **Error banner adds runtime cost** — signal computed + template binding on every page that has load errors | LOW | Banner component is tiny (~50 LoC), only renders when `errorBanner()` is non-null (computed signal auto-unwraps null case). |
| R-4 | **Error-message extraction helper unifies API but loses server-provided messages** — если backend не возвращает чистый message | LOW | `extractErrorMessage(err)` waterfall: HttpErrorResponse.error.message → err.message → fallback string `'Неизвестная ошибка'`. Documented in helper JSDoc. |
| R-5 | **`<app-error-banner>` doesn't fit the Paper & Ink design system perfectly** — design drift | LOW | Reuse `--color-destructive`, hairline border tokens, similar to existing toast styles. Visual diff against toast baseline pre/post. |
| R-6 | **List-page error UX may overlap with toast UX** — both fire for similar errors | LOW | Canonical pattern: GET → banner, POST/PUT/DELETE → toast. Documented in helper-guide `docs/add-new-page.md` §"Error handling". |
| R-7 | **BOM TODO removal may uncover additional ghosts** в других schemas | LOW | Audit grep on backend/src/modules/**.schema.ts extends search for TODO → if found, list in TZ-105-extension backlog. Out of TZ-105 scope. |
| R-8 | **TZ-105 is flush-deployed independently of TZ-103 / TZ-104** — race condition if both ship same week | MEDIUM | git-merge protocol: TZ-105.1 + 105.2 → merge to main via PR; wait 24h; verify no conflict with TZ-104 in-flight branches; then TZ-105.3 → PR. |
