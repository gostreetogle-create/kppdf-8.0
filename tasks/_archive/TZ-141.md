# TZ-141: Page Documentation System

**ПРИОРИТЕТ:** 🟡 MEDIUM  
**LAYER:** frontend / docs  
**DEPENDENCIES:** —  
**ESTIMATE:** 3-4 дня на все страницы  
**CONFLICT KEYS:** `docs/pages/*.md`, `frontend/src/app/pages/**/*.page.ts` (JSDoc)

---

## ROLE

Создать систему документации каждой страницы frontend, чтобы:
1. Разработчики и AI-агенты могли быстро понять, что делает страница
2. Не нужно было перечитывать весь код страницы для понимания контракта
3. MCP-сервер (codebase-memory v0.9.0) автоматически индексировал документацию
4. Уменьшить когнитивную нагрузку: «что принимает, куда отправляет, какие диалоги»

## CONTRACT

- Каждая страница → один `.md` файл в `docs/pages/<name>.page.md`
- Единый шаблон: `docs/pages/_template.md`
- В коде каждого `*.page.ts` — короткая JSDoc ссылка `@see docs/pages/<name>.page.md`
- Индекс всех страниц: `docs/pages/README.md`
- MCP автоматически индексирует `.md` файлы → AI-агенты получают страничную семантику

## МОТИВАЦИЯ

**Текущее состояние:** В проекте 18 директорий страниц (~22 page-компонента). JSDoc внутри файлов описывает историю миграций (TZ-104.3, TZ-104.4.2), но НЕ контракт страницы: route, API endpoints, dialogs, services, state signals. Нет единого места, где можно быстро узнать «что делает эта страница».

**Без документации:** каждый раз при работе над страницей нужно перечитывать весь код, искать API-вызовы, диалоги, lookup tables. Это замедляет разработку и увеличивает риск случайно что-то сломать.

**С документацией:** 5 минут чтения `docs/pages/products.page.md` — и понятно: route, API endpoints, dialogs, state, особенности. MCP индексирует → AI отвечает точнее.
