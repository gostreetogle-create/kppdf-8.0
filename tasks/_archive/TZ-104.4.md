Surface context — `materials.page.ts` post-migration (`fc398a2`):
- `[cellTemplates]` input typed as `Record<string, TemplateRef<{ $implicit: unknown }>>` (TZ-104.3 Phase A-extension).
- Page-side ref fields `@ViewChild('photoTpl', { static: true }) photo!: TemplateRef<any>` — `any` escape hatch.
- Helper methods accept `unknown` (`mainPhotoOf(row: unknown): Photo | null` + `as Material` cast internal).
- Search input flow causes **2 httpResource fetches per keystroke** (page reset immediate + 300ms debounced search).

**Two imperfections to address in TZ-104.4:**
1. **Performance**: 2 fetches per keystroke → 1 fetch (single batched re-fire via `effect()`+`listParams=computed`).
2. **Type safety**: `any`-escape hatch throughout pages → typed `TemplateRef<{ $implicit: T }>` with T-flavored page-side template refs consuming it.

## 2. Root cause analysis

1. **Dual fetch** (`.1`): `onSearchInput` synchronously calls `search.onSearchInput(event)` (which updates `searchQuery` and schedules 300ms-setTimeout for `debouncedSearch`) AND `pageSig.set(1)`. The two signals (`pageSig` + `debouncedSearch`) are both read in `listParams = computed(() => ({ page: this.pageSig(), limit, ...(debouncedSearch()) ? { search: debouncedSearch() } : {} }))`. httpResource fires once per signal-dep change inside one CD cycle; across two CD cycles (synchronous input + 300ms-deferred debounce), it fires twice. Single batched fetch would require both page AND search to flip in the SAME CD cycle — currently they don't.
2. **Type loss** (`.2`): Pi-table accepts `Record<string, TemplateRef<{ $implicit: unknown }>>`. `TemplateRef<C>` is **invariant** in C (C is both input and output context). So the page-side `TemplateRef<Material>` would not be assignable to `TemplateRef<unknown>` — neither to `TemplateRef<Material>` if pi-table were `unknown`-keyed. The `any` escape hatch dodges invariance but loses compile-time `let-row` shape. With Angular 20's strong template type inference (`strictTemplates: true` already enabled in this project per the TS2571 errors encountered in materials v2), the principled fix is to type `[cellTemplates]` as `Record<string, TemplateRef<{ $implicit: T }>>` — `T` is the page-row type, propagated through `<app-pi-table<Product>` etc. (already generic over `<T extends Record<string, unknown>>`).

## 4. Sub-task decomposition

### TZ-104.4.1 — Effect-based debounce coordination

**Files (3):**
- `frontend/src/app/pages/materials/materials.page.ts` — replace `onSearchInput`'s `pageSig.set(1)` with `constructor`+`effect()` block. Drop on-search-input page reset.
- `frontend/src/app/pages/products/products.page.ts` — same (post-migration in TZ-104.7, applies here as TZ-104.7 includes `.1`).
- `frontend/src/app/pages/orders/orders.page.ts` — same.

**Pattern (canonical — applied per page):**
```ts
constructor() {
  let bootDone = false;
  effect(() => {
    const _q = this.search.debouncedSearch(); // explicit dep tracking
    if (!bootDone) {
      bootDone = true;
      return;
    }
    untracked(() => this.pageSig.set(1));
  });
}

// onSearchInput simplified — no explicit page reset
protected onSearchInput(event: Event): void {
  this.search.onSearchInput(event);
}
```

**Why this works:**
- Effect's *first* tick reads `debouncedSearch` (initial value `''`), then sets `bootDone = true` and exits. No page reset — preserves initial load's `page=1`.
- Subsequent typing: every 300ms, debouncedSearch flips → effect re-runs → untracked `pageSig.set(1)` writes only the page signal.
- `listParams = computed(() => ({ page: pageSig(), limit, ...?(debouncedSearch()) ? search : {} }))` reads BOTH page and debouncedSearch in the SAME CD cycle. Angular batches the read. httpResource fires ONCE per actual debounced flip with `{page:1, search:'f'}`.

**Why `untracked()`:** Without it, Angular's effect analytics marks `pageSig` as a dependency of the effect itself — creating a write-dep loop (pageSig.set → effect fires → pageSig.set → ...). Using `untracked()` around the write + invoking the read (`_q`) outside `untracked()` breaks the cycle while still reacting to debouncedSearch.

**Sequencing:** `.1` is independent of `.2`. `.1` ships before TZ-104.7 (products migration) so the effect pattern is canonical for ALL subsequent migrations.

## 7. Risks + Mitigations

| # | Risk | Mitigation |
|---|---|---|
| R-1 | `effect()` write-loop (pageSig → effect → pageSig.set) | `untracked()` around the `pageSig.set(1)` write. Read `debouncedSearch` UN-tracked signal access in effect's body depends on ANGULAR — kept outside untracked. |
| R-2 | `bootDone` closure variable not reset if class re-instantiated | Per Angular lifecycle, NgOnInit fires once per component instance; new instance gets fresh `bootDone`. ✓ |
| R-3 | Effect tracking other signals (pageSize, etc.) inside `listParams` causing spurious refires | Effect body reads ONLY `debouncedSearch`. Verified by AC-3 spec. |
| R-4 | Renderer's `untracked()` API not available in older Angular | Angular 17+ adds it; project is on Angular 20 ✓. |
| R-5 | TemplateRef covariance surprises when `T extends Record<string, unknown>` meets `T2` consumer type | Generic T preserves X. Migration of `any`-escape-hatched pages back to typed triggers TS2345 if column key mismatches `keyof T`. Documented in commit body. |
| R-6 | pi-table tests (cellTemplates x2 + localSort x2) need fixture updates | `<TemplateRef<{ $implicit: TestRow }>>` for typed test. 5 LOC update across 2 describe blocks. |
| R-7 | Migration order: `.1` ships before `.2` because `.2` touches pi-table's contract | `.1` is per-page change (low risk). `.2` is pi-table contract change. Sequential. |
| R-8 | Spread across 27+ pages when all migrated | At `.2` impl time, N is the count of already-migrated pages. Each follows a uniform pattern via `code-reviewer` audit. Mechanical sweep. |
| R-9 | Spec test for AC-3 needs to track HttpTestingController per-call (not via `expectOne` since multiple calls would fail) | Use `httpMock.match()` to count requests per URL across timer ticks. Mock setTimeout via `jasmine.clock()` or `fakeAsync + tick(300)`. |
| R-10 | `.2` might expose latent type errors in TZ-104.5's `pi-input` class-name textarea workaround | Not within scope (different TZ). Documented but unblocked. |
