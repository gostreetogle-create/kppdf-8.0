ИСХОДНОЕ СОСТОЯНИЕ
═══════════════════════════════════════════════════════════════

1. **`work-types.page.ts:194` — `createSortState` snapshot bug:**
   ```ts
   protected readonly sortedRows = this.sort.sorted(this.filteredRows(), (r) => {
     const k = this.sort.sortKey();
     if (!k) return null;
     return (r as any)[k];   // ❌ as any escape hatch
   });
   ```
   
   **Bug A — Snapshot construction:** `this.sort.sorted(...)` — это вызов функции, которая создаёт computed SORTED VALUE разово. `this.filteredRows()` — это computed signal call (резолвится в массив Array на момент вызова). `sortedRows` хранит **текущее значение** snapshot, а НЕ реактивную computацию.
   
   Когда filter меняется → `filteredRows()` computed возвращает новый массив, но `sortedRows` остался прежним (snapshot не пересчитан).
   
   Это работает только потому что `this.sort.sort.sorted()` внутри, видимо, делает reactive binding. **ВНИМАНИЕ:** в `orders/contracts/pages` уже переехали на reactive `computed(() => [...].sort(...))` — `work-types` остаётся legacy.

   **Bug B — `as any` cast:** `(r as any)[k]` — TypeScript escape. До TZ-104.4.2 в products были такие же; после миграции в products убрали.

2. **`stock-movements.page.ts:67` — `numeric: true` + `accessor` returns formatted string:**
   ```ts
   {
     key: 'date',
     label: 'Дата',
     sortable: true,
     width: '10rem',
     numeric: true,       // ❌ triggers numeric sort
     accessor: (row) => this.formatDate(row.date),  // ❌ returns "01.07.2026"
   }
   ```
   
   `numeric: true` в pi-table → sort comparator применяет числовое сравнение к `accessor`-результату. `formatDate('2026-07-01')` → `"01.07.2026"`. Если pi-table приводит к number — `Number("01.07.2026")` → `NaN`. Если coerce — sort вернёт неправильный результат. Date rows отсортированы **по непредсказуемому порядку**.

3. **`storage-items.page.ts:60-66` — `[initialSortKey]="'product'` с `accessor` возвращает строку имени:**
   ```ts
   { key: 'product', label: 'Продукт', sortable: true, accessor: (row) => row.product?.name ?? '—' }
   ```
   
   ```html
   <app-pi-table
     ...
     [initialSortKey]="'product'"
     [initialSortDir]="'asc'"
   />
   ```
   
   **Bug C — initialSortKey introspection:** pi-table может интерпретировать `'product'` как ключ field, не accessor. Если accessor используется ТОЛЬКО для отображения (format cell), а sort — на raw property → sort по объекту `[object Object]` (raw `row.product`). Пользователь не осознаёт что сортирует по reference equality, не по имени.

   Требуется явно документировать в pi-table: используется `accessor` И для сортировки, или только для отображения? Сейчас — undefined behavior.

ШАГ 1: Migrate work-types sort на reactive `computed`

1.1. `frontend/src/app/pages/work-types/work-types.page.ts` — заменить:
```ts
// REMOVE
private readonly sort = createSortState<SortKey>('name');
protected readonly sortedRows = this.sort.sorted(this.filteredRows(), (r) => { ... });

// ADD: inline reactive computed (mirror orders/contracts pattern)
private readonly sortKeySig = signal<SortKey | null>('name');
private readonly sortDirSig = signal<'asc' | 'desc' | null>('asc');

protected readonly sortedRows = computed<WorkType[]>(() => {
  const rows = this.filteredRows();
  const key = this.sortKeySig();
  if (!key) return rows;
  const sign = this.sortDirSig() === 'asc' ? 1 : -1;
  return rows.slice().sort((a, b) => compareWorkTypeValues(a[key], b[key], sign));
});
```

1.2. `setSort(key)` handler меняет `sortKeySig` (cycle asc→desc→null).

1.3. `isSortedBy` / `sortIcon` — переписаны inline.

1.4. Drop `createSortState` import. Remove helper function `compareWorkTypeValues` (private).

1.5. Side effects:
- `(click)="setSort('name')"` — template unchanged (handler signature same).
- search filter change → sortedRows автоматически re-computes (cross-reactive).
- `as any` cast убран.
