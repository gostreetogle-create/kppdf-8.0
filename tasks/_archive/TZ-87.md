## 2. Архитектурные решения

### 2.1 Seed strategy

**Решение:** создать `backend/src/common/seed/dev-fixtures.seed.ts` — отдельный seed module через NestJS lifecycle hook `OnApplicationBootstrap`. Активируется ТОЛЬКО если `process.env.NODE_ENV !== 'production'` (defensive guard).

**Idempotency:** `findOne({slug|name})` first → skip if exists. Не используем `upsert` чтобы не триггерить bcrypt rehash для admin user.

**Scope минимум:**
- 1 Organization («KPPDF Demo Corp», ИНН 7701234567, юр. адрес Москва)
- 1 Counterparty («Demo Client LLC», ИНН 7709876543)
- Maintain existing 6 DocTypes (idempotent — skip-if-exists)
- 0 Materials / 0 Products / 0 ProductModules (TZ-83 dependency — out of TZ-87 scope)

**Trade-off:** Альтернатива — seed через `OnModuleInit` per feature module. Rejected: глобальный idempotent seed проще, явно отделён от production seed, не зависит от feature module boot order.

### 2.2 "Create template" UI placement

**Решение:** добавить `<app-pi-button>` + helper text в `frontend/src/app/pages/doc-constructor/builder/builder.page.ts` empty state:
- Когда `templateOptions().length === 0`, показать `pi-dashed-panel` с eyebrow «Нет шаблонов» + button «+ Создать шаблон» + secondary helper «Подсказка: перейдите в Тексты / Таблицы чтобы добавить reusable блоки».
- Click handler: navigate к `/doc-constructor/builder/new` (new lazy route — TZ-87 B.2 sub-task).

**Trade-off:** Альтернатива — добавить Create button в `PiPageHeader` actions slot во всех состояниях. Rejected: page-header actions = persistent; conditional state-driven UI более предсказуема для пользователя.
