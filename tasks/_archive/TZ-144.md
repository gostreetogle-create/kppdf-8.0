## ⚠️ Важное открытие: excluded.dirs — hardcoded

Исследование MCP `codebase-memory-mcp v0.9.0` (2026-07-19) показало:

- **`excluded.dirs`** (docs/, .git, node_modules, vendor...) — **hardcoded** в бинаре
- `config get excluded_dirs` — пустой ответ (ключа нет в конфиге)
- **Скрытые ключи найдены:** `include_pattern` и `index_include` принимаются через `config set` (см. Шаг 1.1)

### Принятые решения

| Решение | Обоснование |
|---------|------------|
| `include_pattern` и `index_include` — **первые кандидаты** для override excluded.dirs | Оба ключа принимаются бинарём через `set/get`, хотя не отображаются в `config list` |
| Если include_pattern/index_include НЕ влияют — **принять, что docs/ вне knowledge graph** | .md не содержит Tree-sitter структур. MCP-клиент читает .md через file context |
| **Не копировать docs/ в другую директорию** | docs/ — канонический путь. Копия создаст путаницу |
| **Не модифицировать бинарь** | Сторонний closed-source (бинарная поставка, без сорцов для сборки) |

### Workaround'ы (если include_pattern/index_include не сработают)

Если MCP применяет `excluded.dirs` до `include_pattern` (exclude побеждает), docs/ не попадёт в граф. Альтернативы:

1. **JSDoc-ссылки уже работают** — `@see docs/pages/products.page.md` (теперь `Полная документация страницы: ...`) в .page.ts файлах. Tree-sitter парсит эти строки как часть кода, делая docs/ достижимыми через граф (как строковые литералы, привязанные к компонентам)
2. **Создать `docs-index.ts`** — файл-манифест с импортами/референсами всех docs/:
   ```ts
   /**
   * Docs manifest — TZ-144. Заглушка для Tree-sitter, чтобы docs/ был
   * виден в knowledge graph через reference chain.
   * @see docs/design-spec.md
   * @see docs/paper-and-ink.md
   * @see docs/pages/README.md
   */
   export const DOCS_MANIFEST = {
     version: '1.0.0',
     generated: '2026-07-19',
   };
   ```
3. **Ключевые .md → TypeScript doc-comments** — критичные описания API/компонентов можно продублировать в JSDoc-блоках соответствующих .ts файлов (уже частично сделано через `Полная документация страницы: ...`)

Порядок применения: JSDoc ✅ (уже есть) → docs-index.ts (опционально) → include_pattern/index_include (если сработают)

## Шаг 1: Включить auto_index + проверить include_pattern

### 1.1 Установить конфиги

```bash
# Включить автоиндексацию
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config set auto_index true

# Скрытый конфиг: паттерн включаемых файлов
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config set include_pattern "*.md"

# Скрытый конфиг: включаемая директория
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config set index_include "docs"

# Проверить
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config list
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config get include_pattern
./vendor/codebase-memory-mcp/bin/codebase-memory-mcp.exe config get index_include
```

### 1.2 Проверить, влияют ли include_pattern / index_include на excluded.dirs

Запустить MCP с индексацией и проверить excluded-список:

```bash
# Запустить с индексацией (через MCP initialize + index_repository)
# см. scripts/check-mcp-index.sh (создать, если нет)
```

Сравнить `excluded.dirs` из ответа `index_repository`:
- Если `docs` ИСКЛЮЧЁН — include_pattern/index_include НЕ влияют (hardcoded побеждает)
- Если `docs` НЕ исключён — конфиги работают, docs/ в графе

## Шаг 3: Обновить start.mjs

В блок MCP auto-start (уже добавлен в TZ-141) добавить:

```mjs
// TZ-144: Enable MCP auto-indexing + hidden configs
const mcpConfigs = [
  ['config', 'set', 'auto_index', 'true'],
  ['config', 'set', 'include_pattern', '*.md'],
  ['config', 'set', 'index_include', 'docs'],
];
for (const args of mcpConfigs) {
  const r = spawnSync(mcpBinary, args, { stdio: 'pipe', timeout: 5000 });
  if (r.status !== 0) {
    console.warn(`[MCP] ${args.join(' ')} failed (non-fatal)`);
  }
}
```
