## 1. Контекст и обоснование rollback

TZ-93 (shipped 2026-07-11) introduced 3 utility classes:
- `.pi-tech-label` (10px monospace tech label, AAA contrast)
- `.pi-corner-marks` (8px L-shaped corner marks via `::before/::after`)
- `.pi-dashed-panel` (1px dashed border for empty states)

**Проблема:** `.pi-corner-marks` is at risk of reading as **"1990s hacker terminal"** / cyberpunk chrome rather than editorial architectural precision. The 8px L-shapes are a hallmark of:
- Sci-fi terminal UIs (matrix brackets, "ACCESS GRANTED" frames)
- Brutalist design vocabulary (which the project explicitly REJECTED in the TZ-93 spec — see the "What we REJECTED" list)
- Engineering CAD viewers (corner anchors for selection rectangles)

**Why this matters:** TZ-93 spec already declared "brutalist" elements as REJECTED (0px radius, 2px offset shadow, JetBrains Mono everywhere, charcoal primary). The 8px L-shapes — even though smaller than full terminal brackets — still belong to the same visual vocabulary. The JSDoc usage gates ("only empty states, structural cards, showcase panels") were supposed to prevent 1990s risk, but:

1. **TZ-94 planned adopters** (PiEmptyState + PiEmptyTile) put `.pi-corner-marks` in very visible locations (table empty state rows, catalog tile placeholders) — these are exactly the places where editorial restraint matters most.
2. **Conditional application** (`sizePx() >= 80` for PiEmptyTile) was a workaround, not a fix — it just hid the problem in smaller tiles, not solved the aesthetic concern.
3. **The TZ-93 spec's own "REJECTED from brutalist source" list** explicitly excluded corner-style architectural elements (the 0px-radius-everywhere is the most obvious comparison). Keeping `.pi-corner-marks` while rejecting the rest of the brutalist vocabulary is inconsistent.

**Decision:** Drop `.pi-corner-marks` entirely. The 2 surviving utilities (`.pi-tech-label` + `.pi-dashed-panel`) deliver the full architectural feel (mono metadata + dashed empty states) without visual noise.

## 9. Why this is a "rollback" not a "deprecation"

A deprecation would mark `.pi-corner-marks` as `@deprecated` in JSDoc, keep the CSS, and let existing adopters migrate gradually. This TZ-93.1 is a **hard rollback**:
- The CSS is deleted, not annotated.
- There are 0 production adopters (only the playground fixture used it).
- The TZ-94 spec was the only planned adopter, and it's updated to not use the utility.
- TZ-95 is unaffected because it documents the 2 surviving utilities, not the rolled-back one.

A deprecation would only make sense if `.pi-corner-marks` had external adopters (it doesn't — no published components, no npm package). Hard rollback is the right call.
