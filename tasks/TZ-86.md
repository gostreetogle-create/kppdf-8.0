## 1. Цель

Создать **универсальный конструктор документов**, позволяющий собирать шаблоны документов (договоры, КП, акты, спецификации, …) из заранее заготовленных кирпичиков:

```
┌─────────────────────────────────────────────────────────────────────┐
│                      /doc-constructor/builder (canvas)              │
│                                                                     │
│  ┌─ Инструменты (left pane) ─┐  ┌─ Canvas (center) ─┐  ┌─ Inspector │
│  │ ─ Данные (live API):       │  │                   │  │ (right)    │
│  │   • Контрагент dropdown    │  │ Логотип          │  │            │
│  │   • Организация dropdown   │  │ ─────────────    │  │ Конфиг     │
│  │   • Продукция dropdown     │  │ Заголовок        │  │ выбранного  │
│  │   • Цены/сроки/условия…    │  │ Текст-блок A     │  │ блока.      │
│  │ ─ Тексты (read-only):      │  │ Таблица-блок 1   │  │            │
│  │   • "Стандартные условия…" │  │ Данные (Контраг.) │  │            │
│  │   • "Реквизиты сторон"     │  │ Текст-блок B     │  │            │
│  │   • "+ Draft" (literal)    │  │ …                │  │            │
│  │ ─ Таблицы:                 │  │ Подвал + подпись │  │            │
│  │   • "Спецификация товаров" │  └───────────────────┘  └────────────┘
│  │   • "Калькуляция"          │

│  │   • Фоновое изображение    │

└─────────────────────────────────────────────────────────────────────┘
```

## 2. Архитектурные решения (зафиксированы)

### 2.1 Reuse vs Fresh (audit verdict)

| Существующий артефакт (kppdf-8.0) | TZ-86 судьба | Обоснование |
|---|---|---|
| `DocumentTemplate` schema + service + controller | ✅ **REUSE** | Уже хранит name/description/organizationId/docTypeId/pageSize/backgroundImage/backgroundOpacity/version — это именно нужная нам «обёртка» для шаблона документа. |
| `TemplateBlock` schema (blockType: header/text/table/image/signature) | ✅ **REUSE** | Наш атомарный кирпичик на canvas. Schema почти готова, нужны правки ниже. |
| `DocumentTableType` (meta-classifier шаблонов таблиц) | ✅ **REUSE** | Шаблон таблицы = tag/category для структурирования. |
| `DocType` (метатип документа: «Договор», «КП», «Акт») | ✅ **REUSE** | DocumentTemplate.docTypeId уже ссылается. |
| `DocumentBuilder` (фантомный модуль, service.ts отсутствует) | 🆕 **CREATE properly** | В kppdf-8.0 только папка; в kppdf-7.0 были ContractBuilder-доки. Переписываем с нуля — TypeScript service для вычисляемого fill-in данных. |
| Frontend `contract-builder/*` (orphan) | ❌ **DELETE** | Legacy из kppdf-7.0 (Angular-module syntax), не подключен в app.routes.ts. Создаём fresh. |
| `frontend/src/app/shared/services/document-template.service.ts` (legacy) | ❌ **DELETE** | Будет заменён свежим `pi-document-templates.service.ts` (silent-http + сигналы). |

### 2.2 Schema extensions (TZ-86A-F.1)

Schema `TemplateBlock` нужно расширить — добавить поле `dataBinding?` для маппинга к live data sources:

```ts
// TemplateBlock (existing) + new:
@Prop({ type: Object })
dataBinding?: {
  source: 'organization' | 'counterparty' | 'product' | 'contract' | 'order' | 'static';
  // for source: 'static'        → value: string
  // for source: 'organization'  → field: 'name' | 'inn' | 'address' | 'contact' | ...
  // ... see TZ-86 §4.B data binding picker
  field?: string;
  value?: string;
};
```

Также: добавляем `TemplateBlock.kind: 'data' | 'literal'` — `data` блоки подтягиваются через lookup, `literal` — фиксированный текст.

### 2.3 Data registry vs hardcoded (extensibility)

Frontend **никогда** не hardcode'ит dropdown options для поставщиков/организаций. Все списки идут из существующих API:
- `/api/organizations` · `/api/counterparties` (контрагенты) · `/api/products` · `/api/materials` · `/api/work-types`
- Backend поддерживает field projection (`GET /api/organizations/:id?fields=name,inn,address`) — при необходимости добавляем в TZ-86A.

Новые типы данных (новые entites) добавляются просто появлением в БД — конструктор автоматически подхватывает через registry endpoint `/api/registry/data-sources`.

### 2.4 Layout / UX blueprint (canonical 3-pane, decided)

| Панель | Содержимое | Width |
|---|---|---|
| **Left (Инструменты)** | Tab1: Данные (Organ, Counterparty, Product, Material, WorkType — list of lookup-able fields) · Tab2: Тексты (list of saved text blocks) · Tab3: Таблицы (list of saved table templates) · Tab4: Декорации (фоновое image) | 280px, collapsible drawer ↔ |
| **Center (Canvas)** | Live render документа: открывается как «Page» A4 (paper). Page breaks. Block list с drag-handle (переупорядочение). Click block → selected. Inline edit в canvas для text/literal блоков. | flexible flex-1, max-width 1000px |
| **Right (Inspector)** | Конфиг выбранного блока: data-source picker, content field, font-size, alignment, color. Empty state когда ничего не выбрано. | 320px, collapsible |

Rationale: 2-pane (single drawer) wastes vertical space; full-screen single canvas no inspector. 3-pane is canonical pattern (Notion, Figma, Webflow).

### 2.5 Atomic commit pattern

Как TZ-83: 6 атомарных commits, каждый компилируется независимо. Frontend фичи могут опираться на backend endpoints, НЕ наоборот.

### 2.6 Frontend-only ↔ Backend Data

| Источник | Реализация |
|---|---|
| Текстовые блоки (свободный текст, шаблонные фразы) | Backend: `TextBlock` schema (NEW), CRUD endpoints + registry `/api/text-blocks`. Frontend candidates dropdown. |
| Табличные шаблоны (column presets) | Backend: `TableTemplate` (NEW schema, distinct from DocumentTableType — последний это meta-classifier). Schema: `name, columns:[{key,label,type,format?}], sampleRows?: [[cell…]]`. CRUD + `/api/table-templates/:id/preview`. |
| Live data (organisations/products/etc) | Frontend читает existing endpoints. **Никакого нового schema** для live data — конструктор только знает, КАК их рендерить внутри document block. |
| Background image | Backend `DocumentTemplate.backgroundImage: string[]` (existing) + upload endpoint `/api/document-templates/:id/upload-background`. |
| Document preview | Backend `/api/document-templates/:id/preview` → returns rendered HTML (basic — Nunjucks/Markdown inline). TZ-86D implements. PDF export — out of TZ-86 scope (separate TZ). |

### 2.7 Tech stack for canvas (decision)

Canvas renders DOM-based preview, НЕ iframe. Используем Tailwind v4 utilities inline + `@for` для блоков. Drag-reorder через Angular CDK `@angular/cdk/drag-drop` (уже в проекте). NO new heavy dependencies.

### 2.8 Что НЕ делаем (TZ-86 exclusion)

Жёсткий scope guardian. Не входит:
- ❌ PDF/DOCX export (отдельный TZ-XX после MVP)
- ❌ WYSIWYG Microsoft-Word-style редактор с rich-text controls (over-engineering для MVP)
- ❌ Real-time collaboration / WebSocket sync
- ❌ Versioning/history UI (data model versioning уже в DocumentTemplate.version, но UI для rollback — out)
- ❌ Approval workflow для шаблонов (separate TZ)
- ❌ AI-assisted generation (out)
- ❌ Multi-page document editing — MVP = single page A4 layout. Multi-page = separate TZ if user requests.
