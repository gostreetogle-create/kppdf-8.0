# TZ-155: Аудит DTO — @IsObjectId, @IsMongoId, кастомные валидаторы

**Priority**: HIGH  
**Dependencies**: —  
**Estimate**: 3-4h

## Проблема

`grep Pipe` показал только 2 файла с pipe-валидацией — слишком мало для 50+ модулей. Это риск:
- DTO без валидации могут пропускать некорректные данные
- `@IsObjectId()` может отсутствовать на ID параметрах
- Нет проверки enum значений для type-полей
- MongoDB может получать невалидные ObjectId → 500 ошибки

## Требования

1. **Аудит** — проверить каждый модуль в `backend/src/modules/`:
   ```
   for each module:
     - Есть ли DTO/ Create*.dto.ts?
     - Есть ли @IsObjectId() на _id / templateId / parentId?
     - Есть ли @IsEnum / @IsIn на type-полях?
     - Есть ли @IsOptional() для опциональных полей?
   ```
2. **Создать** кастомные декораторы, если отсутствуют:
   - `@IsObjectId()` — проверка `Types.ObjectId.isValid()`
   - `@IsOptionalObjectId()` — опциональный ObjectId
3. **Добавить** в каждый DTO:
   - `class-validator` декораторы
   - `@ApiProperty()` (swagger)
4. **Добавить** global validation error handler в `HttpExceptionFilter`:
   - Человеко-читаемые сообщения на русском для валидационных ошибок
   - Не показывать технические детали (безопасность)

## Acceptance Criteria

- [ ] Все DTO имеют базовые class-validator декораторы
- [ ] Все ID поля имеют `@IsObjectId()`
- [ ] Все enum поля имеют `@IsEnum()` или `@IsIn()`
- [ ] Валидационные ошибки возвращают читаемые сообщения
- [ ] `POST /api/*` с невалидным ObjectId → 400, не 500
