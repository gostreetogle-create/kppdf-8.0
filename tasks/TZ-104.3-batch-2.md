## 2. Root cause analysis

1. **Inventory fragmentation**: 7 pages × 1 sub-task each = 7 small commits. Mechanical, low-risk per page, but loses the cross-page pattern reuse that batch-1 gained. Pattern-axis grouping (envelope vs flat-array) lets each sub-TZ-N document ONE pattern and be re-applied verbatim across N pages of that pattern.
2. **Per-page quirks (lookup tables, status cycles, photo galleries)**: Different from materials/products/orders but each is a localized column/cellTemplate decision. Documented per-page inside each sub-TZ-N.
3. **Backend shape verification gap**: For some pages (organizations, work-types, modules), backend service response shape was NOT yet inspected. Implementation MUST verify shape (envelope vs flat-array) BEFORE writing the migration; mismatched assumption produces tsc-errors that block the commit. See §3.3 "Verify before write" gate.

## 3. Architecture decisions

| # | Decision | Verdict | Rationale |
|---|---|---|---|
| 1 | Sub-TZ-N grouping axis | **Pattern-axis (envelope vs flat-array)**, NOT page-axis | Per the user's explicit instruction: "Group pages by backend-shape (envelope vs flat-array) and stamp a TZ-N for each backend pattern". Pattern-axis maximizes recipe reuse; page-axis would duplicate pattern code 7 times. |
| 2 | Number of sub-TZ-Ns | **2 primary (A envelope, B flat-array) + 1 fallback (C specials)** | A covers 6 pages; B covers 1-2 pages; C is empty unless a special-shape page lands mid-batch. Trivial to author C later. |
| 3 | Backend-shape verification before implementation | **Hard gate — verify each page's service before writing the migration** | Mismatched shape assumption = tsc errors that block commits. Inventory above is a best-effort classification; implementation agent MUST re-verify the actual `list()` method return type before each page migration. |
| 4 | Page-set assignment per sub-TZ | **`.A` envelope = organizations, work-types, modules, dictionaries, stock-movements, storage-items. `.B` flat-array = contracts (+ warehouses if not out of scope).** | Per inventory §1. |
| 5 | Migration recipe source-of-truth | **Embed the recipe inline in §12 of this spec** (mirror materials/products/orders + TZ-104.4 pattern) | A separate `docs/pi-table-migration-recipe.md` was a prior followup; embedding here keeps batch-2 self-contained. The separate recipe doc can be author-extrapolated from §12 if/when batch-3 lands. |
| 6 | Sub-TZ execution ordering | **`.A` first (6 pages, larger pattern), then `.B` (1-2 pages, smaller pattern)** | `.A` reuses the products/materials pattern verbatim — minimal new code. `.B` reuses the orders pattern — same minimal new code. Both have post-batch-1 / post-TZ-104.4 recipes in §12. |
| 7 | Per-page commit granularity | **1 atomic commit per page** (not batched 2-3 per commit) | Per-page commits preserve the clean diffstat + revert-safety of batch-1's "commit 1/3, 2/3, 3/3" pattern. Each commit goes through tsc + code-reviewer independently. |
| 8 | Inventory accuracy | **Mark each page as `assumed envelope` / `confirmed envelope` / `confirmed flat-array` / `TBD`** | Confirmed pages (units, stock-movements, storage-items, contracts, warehouses) have shape verified against service source. Assumed pages (organizations, work-types, modules) need implementer verification. |

## 12. Migration pattern summary (canonical recipe)

### Pattern A — Envelope (server-side paginated + sorted)

Copy from `frontend/src/app/pages/products/products.page.ts` post-TZ-104.4 (`dd88b42`).

**Template structure (per page):**
```html
<app-pi-table
  [data]="data()"
  [columns]="cols"
  [loading]="loading()"
  [total]="total()"
  [page]="page()"
  [pageSize]="pageSize"
  [emptyMessage]="emptyMessage()"
  [ariaLabel]="'Список ...'"
  [cellTemplates]="cellTemplates"
  [rowActions]="rowActionsTplBinding"
  [localSort]="false"
  [initialSortKey]="'name'"        <!-- page-specific default -->
  [initialSortDir]="'asc'"
  (pageChange)="onPageChange($event)"
  (sortChange)="onSortChange($event)"
>
  <!-- cell template (e.g. lookup) -->
  <ng-template #someTpl let-row>...</ng-template>
  <!-- row actions -->
  <ng-template #rowActionsTpl let-row>
    <app-pi-row-actions [row]="row" .../>
  </ng-template>
</app-pi-table>
```

**Class structure (per page):**
```ts
export class OrganizationsPage implements OnInit {
  // Constants
  protected readonly pageSize = 50;
  type SortKey = 'name' | 'status' | 'createdAt'; // page-specific
  // ... lookup tables, signals, etc.

  // List signal wiring
  private readonly listParams = computed(() => {
    const sortKey = this.sortKeySig();
    const sortDir = this.sortDirSig();
    return {
      page: this.pageSig(),
      limit: PAGE_SIZE,
      ...(this.search.debouncedSearch() ? { search: this.search.debouncedSearch() } : {}),
      ...(sortKey && sortDir ? { sortBy: sortKey, sortOrder: sortDir } : {}),
    };
  });
  protected readonly listRes = httpResource<...>(() => ({
    url: `${this.baseUrl}/<endpoint>`,
    params: this.listParams(),
  }));

  // Typed ViewChild (drop-`any` from TZ-104.4)
  @ViewChild('someTpl', { static: true })
  private readonly someTplRef!: TemplateRef<{ $implicit: Organization }>;
  // ... more
  protected cellTemplates: Record<string, TemplateRef<{ $implicit: Organization }>> = {};
  protected rowActionsTplBinding: TemplateRef<{ $implicit: Organization }> | null = null;

  // Handlers
  protected onSortChange(event: { key: string; dir: SortDirection }): void {
    // boundary cast: pi-table emits key: string, page's SortKey is union
    this.sortKeySig.set(event.dir === null ? null : (event.key as SortKey));
    this.sortDirSig.set(event.dir === null ? 'asc' : event.dir);
    this.pageSig.set(1);
  }
}
```

**Class structure (per page):**
```ts
export class ContractsPage implements OnInit {
  // ... no listParams (backend doesn't paginate)
  // ... but does have sortKeySig/sortDirSig, filteredRows, sortedRows, paginatedRows

  protected readonly listRes = httpResource<Contract[]>(() => ({
    url: `${this.baseUrl}/contracts`,
  }));

  // Reactive computed (per orders v4 fix):
  protected readonly filteredRows = computed<Contract[]>(() => {
    const rows = this.data();
    const q = this.search.debouncedSearch().trim().toLowerCase();
    if (!q) return rows.slice();
    return rows.filter((c) => /* page-specific matcher */);
  });

  protected readonly sortedRows = computed<Contract[]>(() => {
    const rows = this.filteredRows();
    const key = this.sortKeySig();
    if (!key) return rows;
    const sign = this.sortDirSig() === 'asc' ? 1 : -1;
    const accessor = accessorFor(key);  // page-specific
    return rows.slice().sort((a, b) => compareValues(accessor(a), accessor(b), sign));
  });

  protected readonly total = computed<number>(() => this.sortedRows().length);
  protected readonly paginatedRows = computed<Contract[]>(() => {
    const all = this.sortedRows();
    const start = (this.pageSig() - 1) * PAGE_SIZE;
    return all.slice(start, start + PAGE_SIZE);
  });
}
```

- All helper internals: NO `as <Row>` casts (template type-inference provides the type).
- All `<ng-template let-row>` references: `row.X` directly typed.
