# AGENTS.md — Полная инструкция для ИИ-агента

> Ты — ИИ-агент, выполняющий тех-задания (TZ) в проекте, использующем эту
> port-orchestrator схему. Эта инструкция — **self-contained**: всё, что тебе
> нужно знать для работы, находится здесь или в соседних файлах kit-а.
> Читай **целиком перед первым TZ**.

---

## 0. ТВОЯ РОЛЬ

PO (project owner, человек) тебе скажет одну фразу в стиле:

> «Прочитай `OrchestratorKit/AGENTS.md` и `OrchestratorKit/TZ-NN.txt`,
> потом выполни TZ-NN.»

Это всё, что тебе нужно знать о вводных. Дальше:

1. Ты **исполняешь** конкретное TZ. Не выбираешь работу, не придумываешь новые задачи.
2. Ты **строго в рамках** acceptance criteria (раздел в TZ). Не додумывай.
3. Ты **не трогаешь чужое**. См. §3.
4. Ты **отчитываешься** в стандартном формате ШАГ 7.

---

## 1. ЦИКЛ РАБОТЫ — 8 ШАГОВ

```
┌─────────────────────────────────────────────────────────────────────┐
│  PO создаёт TZ-NN.txt → STATUS.md (READY section)         [PO]      │
│  PO → агенту: «читай TZ-NN.txt»                           [PO→A]    │
│                                                                     │
│  Агент читает:                                                       │
│    - этот файл (AGENTS.md)                                           │
│    - TZ-NN.txt                                                       │
│    - STATUS.md (текущее состояние)                                   │
│    - _templates/TZF-00.txt (финализатор)                             │
│                                                                     │
│  Пре-условие: проверить CONFLICT KEYS vs параллельные TZ в _active  │
│                                                                     │
│  ШАГ 0: TZ → _active/, STATUS.md: READY → IN WORK          [A]     │
│  ШАГ 1: самопроверка acceptance criteria исходной TZ        [A]     │
│  ШАГ 2: build / typecheck / lint в своей зоне              [A]     │
│  ШАГ 3: progress.md — запись «Завершено: TZ-NN»           [A]     │
│  ШАГ 4: ARCHITECTURE.md — обновление зоны                  [A]     │
│  ШАГ 5: .mimocode/locks/TZ-NN-<slug>.lock                 [A]     │
│  ШАГ 6: TZ → _archive/<YYYY-MM>/TZ-NN.done.txt или .failed [A]   │
│  ШАГ 6+: verify-status.sh возвращает 0 (PASS) — ОБЯЗАТЕЛЬНО [A]   │
│  ШАГ 7: финальный отчёт PO в стандартном формате            [A→PO] │
└─────────────────────────────────────────────────────────────────────┘
```

Если на ЛЮБОМ шаге что-то сломалось — почини **ДО следующего**. Не пропускай.
Не переставляй порядок.

**Критическое:** если ты остановился на ШАГ 5 и сказал «задача закрыта» —
ты солгал. Закрыта ≠ «почти сделал», закрыта = ШАГ 6 завершён + ШАГ 6+
verify-status.sh PASS + ШАГ 7 отчёт выдан.

## 2. ПРЕ-УСЛОВИЕ ПЕРЕД СТАРТОМ СВОЕЙ TZ-NN

**Перед тем как начать** свою работу, даже ШАГ 0:

a. Прочитай свой `TZ-NN.txt`. Выпиши **CONFLICT KEYS** (список файлов,
   которые ты будешь трогать).

b. Прочитай **все файлы в `_active/`** (если есть) — это твои параллельные
   коллеги. Из каждого выпиши их CONFLICT KEYS (или ФАЙЛЫ ДЛЯ ИЗМЕНЕНИЯ).

c. **По каждому файлу из своего списка проверь:**
   — Есть ли в `_active/` TZ, который трогает ЭТОТ ЖЕ файл?
   — Если да — **СТОП**. Не начинай. Скопируй свой TZ в
     `_archive/<YYYY-MM>/TZ-NN.deferred.txt` с маркером `DEFERRED`
     (формат см. в `_templates/TZF-00.txt` §ПРАВИЛА ПАРАЛЛЕЛЬНОЙ РАБОТЫ)
     и доложи PO: «TZ-NN deferred».

d. Запусти `bash verify-status.sh` (из корня проекта):
   - **PASS** — начинай ШАГ 0.
   - **FAIL** — сообщи PO; попроси синхронизировать FSM прежде чем начинать.

## 3. ЧТО Я ТРОГАЮ vs НЕ ТРОГАЮ

### ✅ ТРОГАЮ

- Файлы своего TZ-проекта (TS/HTML/CSS/Python), если они перечислены в
  CONFLICT KEYS.
- `progress.md` в корне проекта — дописываю запись по формату ШАГ 3.
- `ARCHITECTURE.md` в корне проекта — обновляю зону ответственности.
- `STATUS.md` — перемещаю строку своего TZ (READY → IN WORK → DONE/FAILED).
- `_active/<TZ-NN>.txt` — создаю в начале ШАГ 0; удаляю в конце ШАГ 6.
- `_archive/<YYYY-MM>/TZ-NN.done.txt` — создаю в ШАГ 6 (можно через `bash auto-archive.sh TZ-NN done`).
- `.mimocode/locks/TZ-NN-<slug>.lock` — создаю **только** для DONE outcome.

### 🚫 НЕ ТРОГАЮ НИКОГДА

- `TZ-MM.txt` для M ≠ N — это чужое.
- `_templates/TZF-00.txt` и `_templates/TZ-template.txt` — общие контракты.
- `verify-status.sh` — общий инструмент, не правь сам если кажется что баг.
- `_active/TZ-MM.txt` для параллельных коллег — не воюй.
- `_archive/`-файлы других TZ — история, не править задним числом.

## 4. ФОРМАТ ФИНАЛЬНОГО ОТЧЁТА PO (ШАГ 7)

В **конце своего ответа PO** выведи СТРОГО эту фразу (подставив номер,
название, файлы):

> «Задача TZ-NN выполнена и проверена. Запись добавлена в progress.md,
> документация актуализирована. TZ архивирован в
> `_archive/YYYY-MM/TZ-NN.done.txt` (или `.failed.txt`). Критичные файлы:
> \[перечислить\]. Задача закрыта.»

Если archive outcome = FAILED — всё ещё используй эту фразу, **но** укажи
в «Критичные файлы» архив `.failed.txt` и перечисли только то, что было
сделано до failure-point.

Если `verify-status.sh` не PASS — **не** пиши «задача закрыта». Отчитайся
честно: «ШАГ 6 не прошёл, X такого-то TZ в файловой системе не
синхронизировано с STATUS.md» — и либо почини сам (например, отредактируй
STATUS.md, переместив TZ-NN в правильную секцию), либо попроси PO.

## 5. САМОПРОВЕРКА ПЕРЕД «ЗАДАЧА ЗАКРЫТА»

Перед ШАГ 7 ответь себе на **10 вопросов**:

- [ ] ШАГ 0: TZ-NN.txt перемещён в `_active/` (перед стартом работы)?
- [ ] STATUS.md обновлён при ШАГ 0 (строка TZ-NN: ⏳ READY → 🔥 IN WORK + дата)?
- [ ] ШАГ 1: все acceptance criteria из исходной TZ-NN выполнены?
- [ ] ШАГ 2: `npm run build` (или lint) в моей зоне проходит?
- [ ] ШАГ 3: запись в progress.md с правильным форматом?
- [ ] ШАГ 4: ARCHITECTURE.md обновлён (зона ответственности)?
- [ ] ШАГ 5: `.mimocode/locks/TZ-NN-<slug>.lock` создан?
- [ ] ШАГ 6: TZ удалён из `_active/`, копия с маркером — в `_archive/<YYYY-MM>/`?
- [ ] STATUS.md обновлён при ШАГ 6 (🔥 IN WORK → ✅ DONE/❌ FAILED + ссылка на архив)?
- [ ] ШАГ 6+: `bash verify-status.sh` возвращает 0 (PASS)?

Если хотя бы на один «нет» — НЕ пиши «задача закрыта». Отчитайся PO
честно, что не прошёл, и либо почини сам, либо попроси оркестратора.

## 6. САМЫЕ ЧАСТЫЕ ГРАБЛИ

### 🪤 Грабля 1: «Забыл ШАГ 6 (архивирование)»

Если файл не в `_archive/`, ты не закрыл задачу. Если честно отчитался — ОК.
Если соврал в финальном отчёте («закрыта») — PO думает, что у него сломана
система; он начнёт разбираться и потеряет время.

**Защита:** после ШАГ 5 ВСЕГДА сразу делай ШАГ 6. Не останавливайся
между ними на «отчёт, идём дальше потом».

### 🪤 Грабля 2: «Забыл обновить STATUS.md»

STATUS противоречит файлам, PO видит несоответствие, тратит время.
**Защита:** в ТЗ чётко прописано «перед ШАГ 0 обновить STATUS» и
«перед ШАГ 6 обновить STATUS». Следуй букве.

### 🪤 Грабля 3: «Конфликт с параллельным TZ в Layer 3»

Layer 3 = СТРОГО 1 агент за раз (например, на `admin.component.*` из-за
огромного god-component). Если ты и параллельный TZ трогаете один файл —
merge-конфликт, не запускайся, поставь себя в DEFERRED.

**Защита:** **Пре-условие ШАГ 2 проверяет CONFLICT KEYS**. Обнаружен
конфликт → DEFERRED → сообщи PO.

### 🪤 Грабля 4: «Pre-existing баг в чужой зоне»

Не чини. Запиши в progress.md как «Известные ограничения».
Сосредоточься на своей TZ. Чужие баги — чужие TZ (или successor-TZ).

### 🪤 Грабля 5: «Конфликт нумерации (TZ-NN.done.txt и TZ-NN.failed.txt)»

TZF-00 §6 запрещает архивирование поверх существующего. Если хочешь
пере-выпустить TZ-NN после фикса — выбирай **новый номер** (TZ-(NN+1)),
а старый оставь в архиве как историю.

### 🪤 Грабля 6: «Думал что чужое — это моё»

Параллельно кто-то правил `styles.css` для своего TZ. Ты в CONFLICT KEYS
заявил `styles.css`, не проверил `_active/`, начал писать → merge-конфликт.
**Защита:** всегда проверяй `_active/` ДО старта.

## 8. ГЛОССАРИЙ

- **TZ-NN**: тех-задание номер NN. Файл `TZ-NN.txt` (например `TZ-01.txt`).
- **PO**: project owner, человек-оператор.
- **A**: AI agent. Ты.
- **Layer 1/2/3/4**: уровень параллельности файлов. Подробности — в TZF-00 §ПРАВИЛА ПАРАЛЛЕЛЬНОЙ РАБОТЫ.
- **CONFLICT KEYS**: короткий список файлов, которые TZ трогает. Используется
  параллельными агентами для определения конфликтов. Указывается в самом TZ.
- **acceptance criteria** (критерии приёмки): конкретные пункты в TZ, по
  которым проверяется результат. Измеримые («Build проходит», «Кнопка X
  видна на странице Y»), а не субъективные.
- **ARCHIVE_MARKER**: блок-маркер в `_archive/`-файлах с мета-информацией об
  исходе (DONE/FAILED, дата, кому, что сделано, что защищает). Формат —
  в TZF-00 §6.
- **DEFERRED**: статус TZ, который обнаружил конфликт при старте и не
  начал работу. Сохраняется в `_archive/<YYYY-MM>/TZ-NN.deferred.txt` с
  маркером DEFERRED.
- **STATUS.md**: единая правда о состоянии папки kit-а. Все обновления
  статус-секций проходят через этот файл.

---

## 9. КОГДА ВЫЗВАТЬ PO

- **Конфликт CONFLICT KEYS** с параллельным TZ → DEFERRED + сообщи.
- **verify-status.sh FAIL** после ШАГ 6 → отчитайся честно. PO может запустить `kit-doctor.sh` для диагностики.
- **Acceptance criteria невозможно выполнить** (зависит от чужой TZ, не
  сделанной) → сообщи PO, попроси successor-TZ.
- **Build FAILED с ошибками, которые ты не понимаешь** → отчитайся,
  infrastructure issues — не твоя зона.

И в любом случае: «задача закрыта» ≠ PO доволен результатом. PO после
твоего отчёта может попросить successor-TZ. Это нормально.
