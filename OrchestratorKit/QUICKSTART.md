# QUICKSTART — 5 шагов от копирования до первого TZ

---

## Шаг 1: Скопировать kit в новый проект (10 сек)

```bash
cp -R /path/to/source/OrchestratorKit ./my-project/OrchestratorKit
```

Или через IDE: перетащите папку `OrchestratorKit/` в корень нового проекта.

После копирования в новом проекте должны появиться:
```
my-project/
└── OrchestratorKit/
    ├── AGENTS.md              ← AI инструкция
    ├── STATUS.md              ← пустая доска
    ├── _templates/            ← TZF-00, TZ-template
    └── verify-status.sh       ← чекер
```

---

## Шаг 2: Подготовить root-файлы проекта (2 мин)

Создайте в корне `my-project/` два файла, которые агенты будут обновлять:
**`progress.md`** и **`ARCHITECTURE.md`**. Формат — см. секцию «7. Где
что лежит» в [AGENTS.md](AGENTS.md).

Минимальные заглушки:

`progress.md`:
```markdown
# Progress Log — <название проекта>

> Журнал прогресса, обновляется после каждого TZ.

---

(будет пополняться)
```

`ARCHITECTURE.md`:
```markdown
# Architecture — <название проекта>

> Корневой архитектурный документ.

## 1. Общая схема
<!-- пусто -->

## 2. Конвенции
<!-- пусто -->

## 3. Зоны ответственности агентов

| Зона | Файлы | Агент-владелец |
|------|-------|--------------|
<!-- пусто -->

## 4. Открытые вопросы / отложенные задачи
<!-- пусто -->

(будет пополняться)
```

---

## Шаг 3: Проверить, что kit работает (5 сек)

В корне нового проекта:
```bash
cd my-project
bash OrchestratorKit/verify-status.sh
```

Ожидаемый output:
```
PASS: STATUS.md синхронизирован с файловой системой (проверено 0 TZ-файлов)
```

Если PASS — kit XOP-ready. Если FAIL — что-то с путями; читайте
[AGENTS.md](AGENTS.md) §6 для граблей.

---

## Шаг 4: Создать первый TZ-01 (3 мин)

1. Скопировать шаблон: `cp OrchestratorKit/_templates/TZ-template.txt OrchestratorKit/TZ-01.txt`.
2. Открыть `TZ-01.txt` в редакторе.
3. Заполнить **обязательные поля**:
   - **Заголовок** (`TZ-01: <короткое название>`)
   - **РОЛЬ АГЕНТА** (одно из готовых или своё)
   - **ЗАВИСИМОСТИ** (обычно «Нет» для первого TZ)
   - **LAYER** (1/2/3/4)
   - **CONFLICT KEYS** (список файлов, которые TZ трогает)
   - **ИСХОДНОЕ СОСТОЯНИЕ** (что есть сейчас)
   - **ЧТО ДЕЛАТЬ** (шаги, 2-7 пунктов)
   - **КРИТЕРИИ ПРИЁМКИ** (измеримые)
4. Сохранить.

---

## Шаг 5: Добавить TZ в STATUS.md + отдать агенту (1 мин)

1. Открыть `OrchestratorKit/STATUS.md`.
2. В секции `## ⏳ READY (готовы к выдаче...)` добавить строку:
   ```
   | TZ-01 | <название> | <CONFLICT KEYS> | <зависит от> |
   ```
3. Открыть ваш ИИ-агент (Codebuff, Cursor, Claude Code).
4. Сказать **ровно одну фразу**:

> «Прочитай `OrchestratorKit/AGENTS.md` и `OrchestratorKit/TZ-01.txt`,
> затем выполни TZ-01.»

5. **Ждать**. Агент сделает 8 ШАГОВ сам и пришлёт финальный отчёт.

---

## ✅ Готово

После ШАГ 7 агента вы увидите:

- В `OrchestratorKit/STATUS.md`: TZ-01 в секции «✅ DONE».
- В `OrchestratorKit/_archive/<YYYY-MM>/TZ-01.done.txt`: архив.
- В `progress.md`: новая запись от сегодня.
- В `ARCHITECTURE.md`: зона TZ-01 в таблице.

Повторите ШАГ 4-5 для следующих TZ. До 4-5 одновременно (см.
[_templates/TZF-00.txt](_templates/TZF-00.txt) §ПРАВИЛА ПАРАЛЛЕЛЬНОЙ РАБОТЫ).

## 🆘 Если что-то пошло не так

| Симптом | Где смотреть |
|---------|--------------|
| Агент говорит «задача закрыта», но её нет в архиве | [AGENTS.md §5 Самопроверка](AGENTS.md) — по всем 10 пунктам |
| STATUS.md показывает одно, а файлы — другое | `bash OrchestratorKit/verify-status.sh` — покажет конкретные несоответствия |
| Агент не понимает что делать | Дай ему ссылку на [AGENTS.md](AGENTS.md) |
| TZ конфликтует с параллельным | [TZF-00.txt](_templates/TZF-00.txt) §ПРАВИЛА ПАРАЛЛЕЛЬНОЙ РАБОТЫ — три варианта реакции |
| Странный архив (TZ-NN.done.txt и TZ-NN.failed.txt одновременно) | КОНФЛИКТ НУМЕРАЦИИ — см. AGENTS.md §6 «Самые частые грабли» |
