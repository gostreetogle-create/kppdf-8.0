TZ-80 — SSR / Hydration · REJECTED
═══════════════════════════════════════════════════════════════════
Date: 2026-07-05 · Status: ❌ REJECTED · Not deferred, not planned.

Decision
─────────────────────────────────────────────────────────────────
SSR is OUT of the project scope. The Angular 20 frontend ships as
a standard CSR-only SPA from `dist/frontend/browser/`, served by the
inline static server in `start.mjs` (TZ-42). No `@angular/ssr`,
no express, no server build target, no hydration providers.

What this means
─────────────────────────────────────────────────────────────────
· frontend/package.json: NO @angular/ssr, no express, no @nguniversal.
· No `main.server.ts` / `server.ts`. No `provideClientHydration()` in
  app.config.ts. No `server` block in angular.json. No `pnpm build`
  emitting `dist/frontend/server`.
· Existing code that mentions "SSR-safe" (e.g. isPlatformBrowser
  guards in TZ-56 toast, TZ-75 ⌘K palette, TZ-77 Theme Editor) is
  RETAINED as defensive scaffolding — those plugins wouldn't crash
  IF SSR got enabled later. The guards cost nothing to keep.
· TZ-79 (axe-core a11y audit) runs against dev :4200 in CSR mode
  and remains independent of this rejection.
· TZ-82 (smoke test) runs against dev :4200 in CSR mode and is
  INDEPENDENT of TZ-80 — see TZ-82.done.txt for new spec.

Future (out of project scope unless reopens)

  - Either path does NOT depend on TZ-30..79. Just TZ-30 (foundation).
